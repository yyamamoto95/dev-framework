create-dev-framework 業務フロー ワイヤーフレーム

scaffolding CLI の導入体験と、導入後の1日1スプリント運用サイクルを見える化したもの。ID の DF-1xx は導入系、DF-2xx は運用系。

版: 0.1.0

この資料の読み方
  • 業務フロー: 画面の縮小図と矢印で「誰が・どの画面で・何をすると・どうなるか」を表します。矢印の上が操作、下がシステムの応答です。
  • ふるまい表: 各フローの下にある「前提・操作・結果」の表は、開発で使うテストシナリオ（Gherkin）と1対1で対応します。
  • 画面カタログ: フロー中の縮小図をクリックすると、その画面の実寸ワイヤーフレームへ移動します。
  • 青い枠のボタン・リンクはクリックできるホットスポットです。クリックすると遷移先の画面へ移動し、移動先が一瞬ハイライトされます。
  • ⚙ の角丸ボックスは画面を持たない処理（バッチ・自動化など）を表します。下のラベルは処理の主体です。
  • ↗ は外部サービスへの遷移（このワイヤーフレームの範囲外）を表します。
  • 黒い枠は端末（CLI）画面です。$ はコマンド、? は対話プロンプトを表します。
  • フローの下の色付きチップは、その操作でデータ（テーブル）がどう変わるかを表します（作成・参照・更新・削除）。詳細は巻末の CRUD マトリクスへ。
  • ①②… の印は画面下の注釈と対応します。

PBI 単位のスプリントサイクル。計測は GitHub Actions が自動で行い、実績は pull 型で導出する。サイクルの終端は次のスプリントの起点に戻る（ループ）。

1⚙/sprint-startAIエージェントPBI に in-progress ラベルサイクルタイム計測が開始2⚙実装 → /commit → /prAIエージェントレビューを経てマージ3⚙GH Actions が自動計測システム/sprint-closeCycle Time / Sprint# / Story Points をボードへ記録4⚙Retro Issue 起票AIエージェント🏁 Try が retro-action として次スプリントの起点に戻る（ループ）

前提導入が完了し、バックログに PBI がある
